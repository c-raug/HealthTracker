# Plan: Migrate HealthTracker from Expo SDK 54 → SDK 55 + adopt dev client for iOS 26 debugging

> **Status:** Not yet implemented — drafted for execution later.
> **Branch this should be implemented on:** a fresh branch off `main` (do not implement directly on the branch this plan is committed to).

## Context

Currently stuck between two needs:

1. **Live debugging**: today the project uses Expo Go on the iPhone (`npm start` → scan QR), which only supports the SDK that ships with the published Expo Go binary.
2. **Real installable build for iOS 26**: SDK 54's iOS host build is missing iOS 26 fixes; only an SDK 55 build runs cleanly on the iPhone. An earlier attempt at `expo prebuild` against SDK 54 hit issues — which is precisely what motivates the upgrade.

Because there is **no paid Apple Developer account** ($99/yr), the iOS path **cannot use EAS Build for iOS** (EAS ad-hoc distribution requires the paid account). Instead:

- Sign with a **free Apple personal team** in Xcode (7-day re-sign cycle — fits a "use it for the week" plan).
- Run `npx expo prebuild --platform ios` to regenerate the `ios/` folder against SDK 55.
- Open `ios/HealthTracker.xcworkspace` and build to the phone via Xcode.

The dev client (`expo-dev-client`) still gets installed — it's what gives the locally-built iOS app the same Metro / hot-reload / DevMenu experience Expo Go provides today. The daily debug loop becomes: build to phone *once* via Xcode, then for the next 7 days `npm start` + open the dev client app, exactly like Expo Go.

The Android side keeps using EAS — `release-android.yml` already builds APKs on tag push (no Apple cost) and that pipeline keeps working. We'll also add an `ios` block to the `preview` EAS profile, so once a paid Apple account ever exists the iOS preview build path is already wired.

## Step-by-step

### Step 1 — Branch hygiene
Cut a new branch off `main` (e.g. `claude/sdk-55-upgrade`). Confirm clean working tree before bumping anything. If a local `ios/` folder exists from the earlier prebuild attempt and contains uncommitted hand edits (custom `Info.plist` keys, signing tweaks, native-module patches), capture those edits *before* Step 5 wipes the folder.

### Step 2 — Bump SDK
```bash
npm install expo@~55.0.0 --legacy-peer-deps
npm install --save-dev babel-preset-expo@~55.0.0 jest-expo@~55.0.0 --legacy-peer-deps
npx expo install --fix          # realigns expo-*, react-native, react-native-*, react
```
The `--fix` step is the heart of the migration: Expo's CLI knows the exact compatible version of every Expo-pinned dep for SDK 55 and rewrites `package.json` for us.

### Step 3 — Install dev client
```bash
npx expo install expo-dev-client
```
Replaces Expo Go for daily debugging.

### Step 4 — Update `eas.json`
Add a `development` profile and an `ios` block to `preview`:

```json
{
  "cli": { "version": ">= 7.0.0" },
  "build": {
    "development": {
      "developmentClient": true,
      "distribution": "internal",
      "ios": { "simulator": true },
      "android": { "buildType": "apk" },
      "env": {
        "NPM_CONFIG_LEGACY_PEER_DEPS": "true",
        "SENTRY_DISABLE_AUTO_UPLOAD": "true"
      }
    },
    "preview": {
      "distribution": "internal",
      "ios": { "simulator": false },
      "android": { "buildType": "apk" },
      "env": {
        "NPM_CONFIG_LEGACY_PEER_DEPS": "true",
        "SENTRY_DISABLE_AUTO_UPLOAD": "true"
      }
    }
  }
}
```

Note `development.ios.simulator: true` — without a paid Apple account this is the only useful EAS iOS target (runs in iOS Simulator). The physical-iPhone path is Xcode-local. The `preview.ios` block is dormant until/unless a paid Apple account is added.

### Step 5 — Regenerate `ios/` and build to iPhone via Xcode (free personal team)
```bash
# Wipe and regenerate against SDK 55:
rm -rf ios
npx expo prebuild --platform ios --clean

# Install CocoaPods deps (run on macOS, not Codespaces):
cd ios && pod install && cd ..

# Open in Xcode:
open ios/HealthTracker.xcworkspace
```
In Xcode:
1. Select the `HealthTracker` target → **Signing & Capabilities**.
2. Set **Team** to the free personal Apple ID team.
3. Bundle ID stays `com.healthtracker.app` (already in `app.json`) — but with a free team, Apple may auto-suffix / require a unique ID; if Xcode complains, change it to something like `com.healthtracker.app.<your-apple-id-prefix>` *only in Xcode*, leaving `app.json` alone (since `app.json` controls the next prebuild's default).
4. Plug in the iPhone, select it as run destination, hit **Run**.
5. On first launch on the phone: Settings → General → VPN & Device Management → trust the personal-team developer profile.

The build will be valid for **7 days** before the personal-team certificate expires — re-run "Run" in Xcode to refresh.

### Step 6 — Daily debug loop (replaces Expo Go)
```bash
npm start
# or, in Codespaces:
npm run tunnel
```
Open the **HealthTracker dev client** app on the iPhone (not Expo Go), tap "Enter URL manually" or scan the Metro QR. Hot reload, console, and the React DevMenu (shake the phone) work identically to Expo Go. Same `npm start` flow, same QR — just a different host app.

When the 7-day signing expires: re-plug the phone, hit Run in Xcode, debug loop continues. Source code changes never require a rebuild — only the certificate expiry does.

### Step 7 — Sanity sweep
- `npm start` and verify Metro boots without red-screen on launch.
- Smoke-test on the iPhone: bottom tab bar renders, draggable lists drag (gesture-handler still wired), drum pickers in `activities.tsx` scroll on iOS, theme toggle works, AsyncStorage hydrates, Sentry doesn't crash on boot. These touch the libs most likely to break across SDKs (`react-native-gesture-handler`, `react-native-reanimated`, `react-native-draggable-flatlist`, `react-native-screens`, `@sentry/react-native`).
- Verify `.github/workflows/release-android.yml` still runs `eas build --platform android --profile preview` cleanly (no eas.json schema changes broke it). Push a throwaway tag like `v0.0.0-test` or trigger manually.

### Step 8 — Rewrite README.md debug + release sections

The README is the primary reference, so it needs to fully replace the old Expo Go assumption with the new flow. Sections to write/replace:

**`## Debugging locally (daily)`**
1. Pull the branch on the Mac.
2. `npm install --legacy-peer-deps`.
3. First-time only: `npx expo prebuild --platform ios --clean`, then `cd ios && pod install && cd ..`, then open `ios/HealthTracker.xcworkspace` in Xcode, set the personal-team signing, plug in iPhone, hit Run.
4. After first install: just `npm start` (or `npm run tunnel` from Codespaces) → open the **HealthTracker** dev-client app on the phone → tap "Enter URL manually" or scan the Metro QR.
5. Hot reload, console, and the React DevMenu (shake the phone) all work. Source code edits never require a rebuild.
6. **Every 7 days**: re-plug the phone, hit Run in Xcode once to refresh the personal-team signing certificate.

**`## Release process`**
- **Android (existing)**: bump `expo.version` in `app.json`, commit, tag `vX.Y.Z`, push the tag. `release-android.yml` runs `eas build --platform android --profile preview` and attaches the APK to a GitHub Release. (Or run `/cr` for the full release-notes-included flow.)
- **iOS preview (week-long sideload)**: re-run Step 5's Xcode flow against the latest `main`. No tagging, no CI — purely local. Expires after 7 days, just rebuild.
- **iOS via EAS (future, requires paid Apple Dev account)**: profile already wired in `eas.json` under `preview.ios`; the day a paid account exists, run `eas build --platform ios --profile preview` and an `.ipa` will fall out.

**`## Why we don't use Expo Go anymore`**
One short paragraph explaining: Expo Go ships with a fixed SDK, so the only way to run the project on iOS 26 was a custom build; the dev client gives the same hot-reload experience as Expo Go but is tied to *this* project's native deps and SDK.

Keep the rest of the README untouched.
